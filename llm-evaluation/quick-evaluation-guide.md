# 快速评分指南

## 📊 评分速查表

### 1. 用户体验满意度 (20%)

| 分数 | 标准 | Doubao 示例 |
|------|------|------------|
| 5 | 超出预期，完全无需人工干预 | - |
| 4 | 满足需求，仅需少量调整 | ✅ 主动探索项目，代码规范 |
| 3 | 基本可用，需要中等修改 | - |
| 2 | 问题较多，需要大量修改 | - |
| 1 | 无法使用，需要重新实现 | - |

**快速判断**:
- ✅ 代码能直接运行 → 4-5 分
- ⚠️ 需要小修改 → 3 分
- ❌ 无法运行 → 1-2 分

---

### 2. 规划执行反馈 (15%)

| 分数 | 标准 | Doubao 示例 |
|------|------|------------|
| 5 | 清晰的步骤规划，每步都有明确反馈 | ✅ 使用 todo_write，详细 console.log |
| 4 | 有步骤规划，大部分有反馈 | - |
| 3 | 有基本规划，反馈不完整 | - |
| 2 | 规划模糊，反馈很少 | - |
| 1 | 无规划，无反馈 | - |

**快速判断**:
- ✅ 先列出步骤 + 每步反馈 → 5 分
- ⚠️ 只有步骤或只有反馈 → 3-4 分
- ❌ 直接给代码 → 1-2 分

## 🔍 问题类型快速识别

### 11 类常见问题

| 问题 | 识别方法 | Doubao 案例 |
|------|----------|------------|
| **幻觉** | 使用了不存在的 API | ❌ 无 |
| **上下文丢失** | 忘记之前提到的信息 | ❌ 无 |
| **指令遵循失效** | 没按要求做 | ⚠️ 缺少错误处理 |
| **死循环** | 重复相同的操作/回答 | ❌ 无 |
| **偷懒** | 用 `// ...` 省略代码 | ⚠️ 部分"模拟" |
| **代码 Bug** | 语法错误、逻辑错误 | ❌ 无 |
| **废话过多** | 大量无用内容 | ❌ 无 |
| **输出中断** | 回答不完整 | ⚠️ 代码被截断 |
| **目标漂移** | 偏离原始任务 | ❌ 无 |
| **分心** | 关注次要问题 | ❌ 无 |
| **其他** | 未归类问题 | ❌ 无 |

**快速判断**:
- 0-1 个问题 → +5 分
- 2-3 个问题 → +3-4 分
- 4-5 个问题 → +1-2 分
- 6 个以上 → 0 分

**Doubao 问题数**: 3 个 → 得分：4/5

## ⚡ 3 分钟快速评分流程

### Step 1: 浏览回答 (30 秒)
- 看是否有清晰的步骤规划
- 看是否有详细的执行反馈
- 看代码是否完整

### Step 2: 检查代码 (1 分钟)
- 检查定位器是否语义化
- 检查是否有固定延迟
- 检查是否有错误处理
- 检查代码是否完整

### Step 3: 运行测试 (1 分钟)
```bash
npx playwright test llm-evaluation/models/[model-name].spec.ts
```

### Step 4: 统计问题 (30 秒)
- 对照问题类型表
- 统计出现的问题数量

### Step 5: 计算得分 (30 秒)
- 使用评分计算器
- 得出总分和等级

## 💡 评分技巧

### 1. 第一印象很重要
- 如果第一眼看到清晰的规划 → 规划执行至少 4 分
- 如果第一眼看到代码规范 → 代码理解至少 4 分

### 2. 关键检查点
- **语义化定位器**: 使用 `getByRole` 等 → +1 分
- **智能等待**: 使用 `waitForLoadState` → +1 分
- **错误处理**: 有 try-catch → +1 分
- **完整输出**: 代码无截断 → +1 分

### 3. 一票否决项
出现以下问题直接降级：
- 幻觉 API → 代码理解最多 2 分
- 代码无法运行 → 用户体验最多 2 分
- 严重偷懒（大量省略）→ 工程完备最多 2 分

### 4. 加分项
- 主动探索项目结构 → +0.5 分
- 提供详细测试报告 → +0.5 分
- 使用工具链管理任务 → +0.5 分
